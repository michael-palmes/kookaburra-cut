# Device models

Bundled glTF device models for the `Device` / `DeviceMockup` / `HeroObject`
toolkit primitives. Three tiers share one contract:

- **`placeholder-phone.glb` (committed).** A generic handset generated by
  `scripts/make-placeholder-phone.py` (headless Blender). Public clones build
  and run against it — regenerate with:

  ```bash
  pnpm assets:placeholder-phone
  ```

- **`licensed/*.glb` (NOT committed).** The photoreal models are **purchased,
  licensed third-party assets**; `licensed/` is gitignored, so they ship inside
  the maintainer's app builds but never in this repository. Files are named by
  fixed UUIDs (not product names) so bundled asset filenames stay
  trade-dress-neutral; when a file is present locally it overrides the
  placeholder at build time (`src/toolkit/device/modelUrl.ts` resolves
  licensed-else-placeholder, and holds the same mapping as the prep script):

  | Catalog id | Licensed file | Rebuild |
  | --- | --- | --- |
  | `iphone-15-pro` | `licensed/6241bad0-f016-4c0f-95c0-9aac0930a6ac.glb` | `pnpm assets:phone` |
  | `iphone-17-pro` | `licensed/e1bfddac-38f7-48a6-adf0-0d0120b7e937.glb` | `pnpm assets:iphone-17-pro` |
  | `macbook-pro-16` | `licensed/b30d3bc4-a66b-4376-95d1-30978b87212c.glb` | `pnpm assets:macbook-pro-16` |

  All three run `scripts/prepare-device-model.sh` (Blender export with a
  per-device corrective yaw, then gltf-transform optimise; needs
  `KOOKABURRA_ASSETS_DIR` pointing at the private vendor-blend folder).
  `pnpm assets:devices` rebuilds the lot. Sanity-check any rebuilt glb with
  `scripts/blender-render-glb-check.py` (renders the glTF +Z view, which must
  be the screen, dead-on) and `scripts/dump-glb-materials.mjs` (material names
  + factors, the colour-override ground truth).

- **`android.glb` (committed).** A generated, unlicensed Pixel-style handset,
  committed directly under its real name (no UUID indirection). The source
  `android.obj` lives outside the repo; rebuild the GLB and colour previews with:

  ```bash
  KOOKABURRA_ANDROID_OBJ=/path/to/android.obj pnpm assets:android
  ```

  The OBJ carries flat MTL colours only, so `scripts/prepare-android-model.py`
  gives the metal/glass/bezel materials PBR values and embeds a small gradient on
  the `screen` material (for the texture-decode guard below), then gltf-transform
  optimises it (palette + simplify OFF). Colours (graphite, black, white) are
  catalog overrides on `frame_metal`/`camera_bar`/lens rings and `back_glass`;
  previews render via `scripts/render-android-previews.py`.

The committed device preview images render the licensed models; they are not
covered by the repository licence and may not be reused outside this project
(see the root `NOTICE.md`).

## Model contract (all tiers)

- **Screen mesh:** one near-planar mesh whose material name matches the
  catalog's `screen.material` for that device (`SCREEN` on the 15 Pro,
  `screen` on the 17 Pro, `SCREEN.001` on the MacBook) — the name is
  per-`DeviceSpec` in code, so no rename inside the vendor file is needed.
  The primitive swaps it for an unlit material showing the scene's media.
- **Screen UVs:** span the display 0..1 in glTF orientation (v=0 at the top).
- **Hidden nodes:** anything that must not render (a baked studio plate or
  render helper) is named in `HIDDEN_NODES` (`src/toolkit/device/models.ts`)
  as three.js sanitises it (spaces to underscores, dots stripped): `BG_Plane`,
  `Bg`, `Cube`, `View_Blocker001`.
- **At least one textured material** — the catalog preload's texture-decode
  guard treats an all-untextured parse as a broken texture pipeline (the
  placeholder embeds a small gradient on `SCREEN` for this).
- Optimised, offline-deterministic: no Draco/KTX2 (they need CDN decoders),
  textures ≤2048, metres scale, front (the screen) faces glTF +Z at identity
  rotation. gltf-transform's `palette` step stays OFF — it fuses materials
  into anonymous names, breaking colour overrides and the screen lookup.
- **Simplify is a fidelity hazard on hard-surface models.** Its defaults
  (borders free, error 0.0001) collapse exactly the fine features product
  models are made of: the MacBook lost its keycap rims and speaker
  perforations to it (264k keycap verts down to 37k). The pipeline now locks
  topological borders and drops the error budget to 0.00001, and the MacBook
  turns simplify off outright (locked borders still creased its spacebar,
  right shift, F3 and F5). Check a rebuilt model's fine detail before
  trusting a size win.
- Catalogue colour variants are per-material `baseColorFactor` overrides
  applied by material name (`src/toolkit/device/catalog.ts`); names missing
  from a model simply don't apply (the placeholder ignores them). Vendors
  using the Blender Metallic BSDF are converted to Principled at export
  (`scripts/blender-export-glb.py`) so their colours survive.
- **Mix Shaders do not exist in glTF.** A vendor material that mixes two
  Principled BSDFs through an image (the MacBook's `PLASTIC Keyboard`, whose
  mask carries the key legends) exports with no texture at all and then
  dedups into a neighbouring plain material, taking its name with it. The
  export script composites the mask into a single base-colour texture so one
  Principled carries it.
- **Perforated grilles need a backing.** The MacBook's speaker grilles are
  holes through the top case, so through them you see the internal shell and
  ports (a busy, inconsistent mess). The export script drops an opaque dark
  quad just under each perforated strip (placement derived from the casing and
  keyboard-tray bounds, so it self-skips any device without them) to give the
  holes a flat, consistent dark field.
